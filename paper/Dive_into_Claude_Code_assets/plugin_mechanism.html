The Plugin Mechanism — one package, many registries

Plugins are not a new runtime primitive. A manifest declares up to ten contribution types; pluginLoader.ts validates and fans each one into the registry that same component would use if installed by hand.

How to read the diagram (left → right):

  • Left (package on disk). A concrete py-dev-kit/ directory. One plugin.json manifest plus contribution subfolders (commands, agents, skills, hooks, mcp, lsp, styles, config). This is what a developer publishes — the "package" side.

  • Center (focal coral). pluginLoader.ts validates the manifest against PluginManifestSchema and then fans the declared components out. The coral arrow from package → loader is the "install" edge — this is the "distribution" side.

  • Right (six runtime registries). Commands + skills → SkillTool meta-tool. Agents → AgentTool definitions. Hooks → hook registry (27 events). MCP servers → MCP client config. LSP servers → LSP config. Styles/channels/settings/userConfig → formatter + config merge.
